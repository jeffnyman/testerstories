Designing and Testing Grue

Written by Jeff Nyman 4 June 2023 Leave a comment

In this post I want to level-set on a particular starting point for the implementation of Grue, from a development and testing point of view. I’ve covered a lot in the previous posts, with code interspersed with narrative. Here we’ll pretty much just focus on code along with the tests.

The Importance of Testing

For those coming to these posts to learn how to build a Z-Machine interpreter, I realize the code-based tests I talk about may be of less importance and may even seem to be a bit of a distraction. However, there’s something important to keep in mind here.

At the point we’re at — and at the point you start developing up to for quite a good amount of time — you won’t really have a working implementation yet. There’s nothing someone, including yourself, could actually test if you distributed the implementation. Yet that doesn’t mean you can’t test at all.

And this is crucial! You need to make sure that you’re not breaking the existing parts (introducing regressions) as you build up more of the functionality. And, as my testability series showed, introducing bugs — both in code and design — is frightfully easy to do, sometimes in quite subtle ways. Thus a cost-of-mistake curve is always worth considering. This means asking yourself “What is the duration from the time I make a mistake to the time I find it?” The longer that duration the more complicated your code becomes.

The Grue Program

As stated from early on in this series, my goal is to write Grue in Python. But I’m keeping the code as simple as possible — which in some cases means as non-idiomatic as possible — in order to make it easy for you to use another language if you choose.

def test_load_zcode_program(zork1_z3, monkeypatch, capsys) -> None:

This, to me, is actually an annoyingly complicated looking test, requiring me to use “capsys” (retrieving stdout and stderr from code) and “monkeypatch” (alter execution in some way). It’s also arguable if this test is really testing anything. As more tests come up later, I’ll have reason to question whether I even need this. For the test-oriented reading these posts, this is a good point to keep in mind about testing and how developers often have to approach it.

class Memory:
    def __init__(self, data: bytes) -> None:
        self.data: bytes = data
 
        self.version: int = self.read_byte(0x00)
        self.release_number: int = self.read_word(0x2)
        self.serial_code: bytes = self.read_bytes(0x12, 6)
 
        self.dynamic: int = 0
        self.static: int = self.read_word(0x0E)
        self.high: int = self.read_word(0x04)
        self.routine_offset: int = self.read_word(0x28)
        self.strings_offset: int = self.read_word(0x2A)
 
        self.pc: int = 0
 
        self._read_starting_address()

This code is primarily reading bits of memory from the loaded zcode program. The helper methods for doing this — read_word() and read_byte() — are defined on the memory as:

def read_bytes(self, address: int, length: int) -> bytes:
    return self.data[address : address + length]
 
def read_word(self, address: int) -> int:
    return (self.data[address] << 8) | self.data[address + 1]

Here’s the _read_starting_address() method which is called when the Memory instance initializes.

Python
1

def _read_starting_address(self) -> None:
    if self.version != 6:
        self.pc = self.read_word(0x06)
    else:
        self.pc = self.read_packed(self.read_word(0x06), True)

This method is really important because it’s what establishes where the interpreter should start reading instructions from. Like the methods for reading bytes and words, the read_packed() helper method is also defined on the memory class:

def read_packed(self, address: int, is_routine: bool) -> int:
    if self.version < 4:
        return 2 * address
 
    if self.version < 6:
        return 4 * address
 
    if self.version < 8 and is_routine:
        return 4 * address + (8 * self.routine_offset)
 
    if self.version < 8:
        return 4 * address + (8 * self.strings_offset)
 
    return 8 * address

Unlike the simple act of reading a byte or a word, a packed address has a specific implementation that I had to read the specification to understand.

All of what we looked at happens as a result of instantiating the Memory. So let’s revisit two of the lines from earlier:

Python
1
2
memory = Memory(data)
memory.details()

The first one is instantiating the memory with the binary data. The memory.details() is just something I implement to log out the information read so far. Running all this on the zork1.z3 program file (used in my test fixtures) gives me:

zcode version: 3
Release number: 88
Serial code: 840726
Static memory: 0x2e53
High memory: 0x4e37
Routine offset: 0
Strings offset: 0
Starting address: 0x4f05

One thing to note and this is a bit of an aside based on the last post. If the routine offset and the strings offset are both 0 — as they are here — then that means that the zcode program does not use packed addresses for routines or strings. This would mean that all routines and strings are located at their absolute addresses in memory.

However, even in that situation, it’s still the case that all routines use a packed address to say where they begin. Thus even if a game’s zcode does not use the routine offset and strings offset fields, packed addresses can still be used to refer to routines and strings in high memory.

instruction: "Instruction" = memory.read_instruction()
instruction.details()

Here’s the read_instruction() method:

Python
1
2
3
4
5
def read_instruction(self) -> Instruction:
    instruction = Instruction(self)
    instruction.decode()
 
    return instruction

Just like I have a Memory class to represent the memory of the interpreter, I’m going to have an Instruction class to represent each instruction that’s encountered.

As I was working on this, I did wonder if having a class instance for each instruction might be a bit too much overhead in terms of execution. Granted, we’re talking about text adventures operating on modern hardware but, still, I was wondering about the efficiency of this. I actually don’t have an answer to that. In my full Quendor application, I don’t think I’m going to use that approach.

For now the main thing is just setting up the instance of the instruction.

class Instruction:
    def __init__(self, memory: "Memory") -> None:

Each instruction has access to the interpreter memory. This will allow me, for example, to call those helper methods to read areas of memory or to perhaps get the version instance variable value.

And that’s about it for the immediate core implementation. In the next post, we’ll get into this line of code:


instruction.decode

Hopefully the above gave you some straight context for implementing a Z-Machine without a lot of the surrounding narrative.

What we did is provide an ingestion mechanic for getting a zcode binary loaded up and its contents read. We then established an “interpreter memory” that will store a bunch of details about the zcode program. We got the location in memory where the first instruction to be executed should be stored. What remains now is the meat of the interpreter which we’ll look at next post.
