Processing in Grue

Written by Jeff Nyman 2 June 2023 Leave a comment

At this point, we’re very close to one of the meatier subjects in terms of creating a Z-Machine implementation, which is the decoding of instructions from the zcode binary. Before we get into that, let’s take a look at tthe overall processing that will need to be supported.

Emulating the Z-Machine basically means emulating a CPU, often just referred to as a processor. The processor at the heart of any computer — even a virtual one — can ultimately understand only the most simplistic of instructions. These instructions, known as “operation codes” (or just “opcodes”) do really simple things. Some examples of “really simple things” are:

  • Moving a single number from memory into a register of the processor.
  • Adding a number already stored in a register to another number.
  • Putting the result from an operation back into memory.

Each opcode is identified by a unique sequence of “on/off switches,” better known as bits. Some of the earliest programmers were quite literally bit flippers. This meant they entered long sequences of 1s and 0s by hand, essentially “flipping” a given bit to one of the two values.

Technically you could argue that’s all programmers do these days as well but the levels of abstraction are so high that what we do now is light years in distance from what we did back when computing was just getting started.

Tools called assemblers were eventually developed and these allowed those programmers to replace 1s and 0s with unique textual identifiers, often referred to as mnemonics because they were meant to indicate what the operation did. For example, “STO” would store a number in memory, “ADD” would perform an addition operation, and so on.

To use a retro example, firing up my Commodore 64, I could enter this:


POKE 53280,0

That’s a statement written in BASIC. The memory address 53280 (0xD020) is used for the background color of characters and graphics of the machine. The above command would set the background to black. What does POKE mean, though? It was just the mnemonic for an instruction that was used to change the contents of any address in memory. A PEEK instruction, by contrast, would provide you with the contents of memory at a given address.

Incidentally, there is no “POKE” equivalent in 6502 assembly language. (The Commodore 64 was based on the 6502 microprocessor and its instruction set architecture.) In that context, it’s not possible to load a memory location with a value within a single instruction. So, to change the color to black, the BASIC statement “POKE 53280,0” actually becomes this:


LDA #$00
STA $d020

The mnemonic “LDA” (Load Accumulator) is used to load a value into the accumulator, and the mnemonic “STA” (Store Accumulator) is used to store the value from the accumulator into a memory location. In this case, “LDA #$00” loads the immediate value of 0 into the accumulator, and “STA $d020” stores the value from the accumulator into the memory location represented by $d020, which corresponds to the chip’s register controlling the background color.

Without the mnemonics, the assembly code snippet would be represented like this:


A9 00
8D 20 D0

How that breaks down is:

  • A9 is the opcode for the “Load Accumulator Immediate” instruction.
  • $00 is the immediate value being loaded into the accumulator (register A). In this case, it’s 00 which represents the value zero.
  • 8D is the opcode for the “Store Accumulator Absolute” instruction.
  • $20 D0 is the memory address where the accumulator value will be stored. In this case, it refers to the memory address $D020, $20 is the low byte, and $D0 is the high byte. Together, they form a 16-bit memory address.

In binary, this would have been:


Load Accumulator Immediate:
A9 00000000

Store Accumulator Absolute:
8D 20D0 00100000 11010000

While these mnemonics were useful aids for programmers, they meant nothing to any computer. It was only that lowest level binary that actually meant anything to a processor. So after writing the program using a system of mnemonics, the programmer had to pass it through the assembler to generate the 1s and 0s the computer needed.

Eventually higher-level languages were created that would let these programmers create their programs at a much greater level of abstraction from the hardware. This allowed them to focus more on the logic of what they were trying to achieve and less on bit twiddling.

The first really complete example of such a language arrived in 1954 and was called FORTRAN. This language is historically interesting (relevant to this project anyway) because it was the language a guy named Will Crowther chose to code a game called Colossal Cave Adventure in. This was back in 1975. Zork would initially pay homage to this game and was arguably written as a successor to it.

Another such language was LISP, which came on the scene around 1958. This one is historically interesting (again, as relevant to this project) in that it was the ancestor of a language created at MIT called MDL. That, in turn, was the ancestor to a language created by Infocom, called ZIL (Zork Implementation Language). ZIL is ultimately what the Zork that most people played was written in.

A program written in ZIL was compiled into a form of bytecode, which is the zcode we’ve been talking about since this series began. This zcode was executed by having it run through another program called a ZIP (Z-Machine Interpreter Program).

At this point in the series, you certainly know that the ZIP is what Grue has to emulate. This is a lot of what we’ve already discussed in these posts.

The Zcode Intersection

What this recap does is let us understand that zcode is sort of the intersection between the bits and bytes described in a previous post and the Z-Machine architecture, which will be described as we continue. Here’s a view of that architecture again just so we keep it in mind.

Zcode is a bytecode compiled language. The native language of the Z-Machine is this assembly language, just as 6502 assembly language was the native language of the Commodore 64.

At this point, I should note that there is a document referred to as the Inform Designer’s Manual, version 4 (often abbreviated as DM4), which serves as documentation for the Inform development system. This document has a few things to say about the Z-Machine architecture and, in that context, the DM4 provides this snippet of code:


"The answer is ", 3 * total + 1;

This is essentially a statement in the high-level language of Inform, which is very different than the ZIL that Infocom was using. An equivalent ZIL might look something like this:


<SETG TOTAL >
<SETG TOTAL <+>>
<TELL "[ The answer is " N .TOTAL ".]" >)>>

However, the nature of the source code matters not at all. What matters is the bytecode. The above logic would be rendered like this:


@print "The answer is ";
@mul 3 total -> x;
@add x 1 -> x;
@print_num x;
@new_line;
@rtrue;

What you have there is assembly code. Assembly languages have commands that are called opcodes. In the above example, those are written with an @ sign in front of them. The values supplied to opcodes, such as 3, 1, and total, are called operands.

Opcodes and operands are going to be really important as we start building the next parts of the Z-Machine implementation.

A zcode program is made up of a sequence of bits and bytes and your interpreter essentially has to loop through those sequences. So let’s talk about that part next.

The Interpretation Loop

As you can see in the visual above, the Z-Machine itself has an architecture and, ultimately, you have to implement that architecture if you want to actually emulate the machine itself. That said, you can consider the architecture part in stages when you are designing your Z-Machine.

The simple part of the architecture, and the first to be implemented, can be thought of as the interpretation loop. Then there’s the more complex part of the architecture, which is the actual emulation of the machine so that the interpretation loop is carried out correctly.

“Carried out correctly,” in this context, means being able to actually play one of the games, like Zork, that run on the Z-Machine.

As far as the simple part, your interpreter has to do the following:

  1. Load a story file that contains zcode.
  2. Get the byte contents of the story file.
  3. Determine which byte the story execution starts at.
  4. Decode the instruction at the starting byte.
  5. Loop through any remaining instructions.

A “story file” is what Infocom called their game files. The byte contents of the file were essentially the starting state of the game. As the game was played, different instructions would be carried out, based on whatever the player did, and that would change the state of the game.

For example, in Zork the player can open a mailbox and take a leaflet out of that mailbox. The player can open a trapdoor and descend into a dungeon whereupon the trapdoor will be closed and locked behind them. All of thse are changes to the state of the game based on player action.

Step 4 above is clearly the most involved of this initial part of the architecture. You essentially have to understand how to determine the nature of instructions, which is where the Z-Machine specification comes in. After you have that part down, then you have to provide an implementation for each instruction. Once again, you’ll turn to the Z-Machine specification to understand how to do that.

All of this is what Grue has to do.

That’s the low-hanging fruit of the architecture you have to implement. Once that bit of architecture is in place, the work comes down to implementing all of the instructions. So, with the above assembly example, you have to provide implementations of @print, @mul, @add, @print_num, @new_line, and @rtrue.

But it gets tricky. I mentioned that some opcodes take operands. Operands can only be constants or variables. So consider this assembly instruction:


@add x 1 -> x;

This instruction adds x (a variable) and 1 (a constant), storing the result of this addition back in x. And this brings up an important point: some opcodes store values and some don’t. So this is a part of the architecture that you need to implement. One part of that, for example, would be the stack.

The reason this is relevant so quickly is because one of the first opcodes you’ll work with in implementing a Z-Machine is the call opcode and, in that context, the stack can have some relevance. For now I’m going to keep this pretty simple. The Z-Machine contains a stack which is accessed with a stack pointer. The stack is basically nothing more than a bucket of values. Your interpreter must write to and read from this bucket by using the stack pointer.

  • Writing to the stack pointer means placing a value on top of the stack.
  • Reading from the stack pointer means a value is taken off the top of the stack.

A trick is that each routine in a zcode story file has its own stack. Some people like to refer to a “call stack” (for all routines) and then each routine itself having its own “evaluation stack.”

Let’s consider another way that things can happen. Consider the following assembly:


@je x 1 ?Xisone;
@print "x not equal to 1";
.Xisone;

The instruction @je means “jump if equal” and what this does is test an operand, in this case x, to see if it’s equal to another operand, in this case 1. If that’s the case, the code will jump to the label called Xisone, effectively skipping the @print instruction.

And this brings up another important point: some opcodes can branch. This results in execution jumping to a different place in the code if some condition turns out to be true.

What the above is showing you is that some of these instructions are more complicated than others. So when building an interpreter, and the implementation of the instructions, you do need to have some understanding of the Z-Machine architecture.

You’ll have to learn, for example, how attributes are dealt with (they can only be true or false; set or unset). You’ll have to learn how properties are dealt with (they can contain values). You’ll have to learn how the object table works (so that you can make sure object manipulation occurs as it should such that object relationships are maintained).

Where to Start?

But where do we start doing all this? Well, the specification says:

The processor begins to run a story file by starting with an empty stack and a PC value set according to some information in the story file’s header. Note that the story file has to be set up with many of the structures in memory, such as the dictionary and the object tree, already created and with sensible contents.

That setting of the program counter (PC) is crucial. But where do you set it to? Again, let’s turn to the specification:

In Version 6, there is a “main” routine (whose packed address is stored in the word at $06 in the header) called when the game starts up.

And:

In all other Versions, the word at $06 contains the byte address of the first instruction to execute.

I have logic on my Memory instance to get this information and set the value of the program counter to it:

def _read_starting_address(self) -> None:

    else:
        self.pc = self.read_packed(self.read_word(0x06), True)

You can see that right at the start this brings up the idea of routines.

Routines

Let’s make sure we have the concept of routines down. According to the specification:

Programs are divided into routines: the Z-machine is always executing a particular routine, the one which the PC currently points inside.

Consider that statement above where read_packed() is used. This is a method we looked at already, but here it is again for reference:

def read_packed(self, address: int, is_routine: bool) -> int:
    if self.version < 4:
        return 2 * address
 
    if self.version < 6:
        return 4 * address
 
    if self.version < 8 and is_routine:
        return 4 * address + (8 * self.routine_offset)
 
    if self.version < 8:
        return 4 * address + (8 * self.strings_offset)
 
    return 8 * address

Important here are those ideas of offsets so let’s consider those briefly.

Routine and Strings Offsets

A routine is a block of code that can be executed by a zcode program. A routine offset refers to the location of a routine within the Z-Machine’s memory. In the Z-Machine’s memory, a routine is preceded by a packed address that specifies its starting location.

As discussed in the previous post, the packed address is a compressed form of a 17-bit pointer that allows the routine to be located anywhere in memory. The routine offset is the number of bytes from the start of the Z-Machine memory to the packed address that precedes the routine.

For example, suppose we have a routine that starts at address 0x3A7C in the Z-Machine’s memory. If the packed address for this routine is stored at address 0x7F8A, then the routine offset would be 0x7F8A - 2 (since the packed address is two bytes long) - 0x3A7C = 0x4412.

The routine offset is useful for interpreters and compilers that need to generate or modify Z-Machine code. By calculating the routine offset, an interpreter can easily find the packed address for a routine, and a compiler can easily update the packed address after generating or modifying a routine.

So the key point here is that all routines in the Z-Machine require a packed address, regardless of whether they are stored in the lower 64K or beyond it. The packed address is used to encode the address of the routine in the zcode program, and the routine offset is used to look up the actual address of the routine in memory at runtime.

The Z-Machine specification says this:

A packed address specifies where a routine or string begins in high memory.

So it sounds like routines themselves can be stored in static memory but where all the routines begin is stored in high memory. And that’s correct. While the routines themselves may be stored in the static memory, the starting addresses of all routines are stored in high memory. This allows the interpreter to easily locate the entry point of any routine.

So what about those “strings”? Well, technically, that takes us a little afield of what we’re talking about here but since it’s in the same context, let’s at least take a brief look.

In the Z-Machine, a “strings offset” is an index into the high memory where the start of the table of strings is located. The Z-Machine specification defines a table of strings that is stored in high memory and is used by the program to store textual data, such as game messages, prompts, and other user interface text.

Like the routine offset, the strings offset is also expressed as a packed address. The packed address for the strings offset is stored at a byte offset in the Z-Machine header, and it follows the same encoding scheme as other packed addresses.

When a Z-Machine program needs to display a string, it typically reads the packed address for the strings offset from the header, unpacks it into a 16-bit address, and then uses that address to locate the start of the table of strings in high memory. From there, it can read the appropriate string based on the index into the table specified by the program.

At this point, the above code from _read_starting_address() will tell you where to point the program counter of your interpreter.

Now that we know where to start execution, we can get into actually decoding the instruction and that’s what we’ll start tackling next.
